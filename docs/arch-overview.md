
# Moon Programming Language: Architectural Overview

This document serves as the comprehensive architectural map of the Moon programming language's C implementation. It is designed to be presented for a final year project defense, explicitly defining the boundaries, responsibilities, and structural paradigms of the language.

The codebase is split into three primary phases: **The Compiler Frontend**, **The Compiler Backend**, and **The Virtual Machine Runtime**, alongside developer tooling such as the native Language Server.

---

## 1. The Compiler Frontend (Lexical & Syntactic Analysis)
The frontend is responsible for translating raw source code strings into a structured Abstract Syntax Tree (AST). It strictly separates lexical tokenization from syntactic parsing.

### 1.1 `scanner.c` / `scanner.h` (The Lexical Analyzer)
The scanner converts a raw string of text into a stream of discrete `Token` objects. 
* **State Management**: It maintains its own state independently of the parser, managing flags like `interpolationDepth` (to handle nested string interpolation like `"Value: ${x}"`) and `preserveComments` (used when the LSP needs to parse comments for documentation hover).
* **Token Definitions**: Defines all 1-character, 2-character, and literal tokens inside the `TokenType` enum.

### 1.2 `parser.c` / `parser.h` (The AST Builder)
The parser consumes tokens from the scanner and builds the Abstract Syntax Tree using a Pratt Parsing (Top-Down Operator Precedence) algorithm.
* **The Parser State**: Maintains the `current` and `previous` tokens, alongside a `panicMode` flag to gracefully recover from syntax errors without cascading failures.
* **The "Blindfold State"**: Uses an `ExpectedLabel` stack to track expected block terminations dynamically, allowing Moon's freeform and flexible syntax (e.g., matching `if` blocks without requiring explicit curly braces, driven by the "Colon Rule").
* **Output**: Produces the master `Node` tree.

### 1.3 `ast.c` / `ast.h` (The Abstract Syntax Tree)
Defines the absolute data structure of the language's grammar.
* **The Master Node**: Every grammatical construct is represented as a `Node` containing its `NodeType` (e.g., `NODE_LITERAL`, `NODE_IF`, `NODE_FOR`), the original line number, and a parent pointer.
* **Payload Structs**: Depending on the type, the node utilizes a C `union` (`as`) to hold its specific payload (e.g., `as.binary` holds `left` and `right` nodes).
* **Dynamic Arrays**: Uses `DEFINE_ARRAY` macros to create resizable arrays for nodes (`NodeArray`).

### 1.4 `sigtrie.c` / `sigtrie.h` (The Phrasal Signature Trie)
Moon features a unique English-like phrasal function calling syntax (e.g., `add $1 to $2`).
* **The Trie DFA State**: Function signatures are stored natively in a Trie (Prefix Tree) data structure. `TrieNode` manages the branches.
* **The Hash Table**: `phrasalTable` registers the root words to start the DFA search.
* **Execution**: During parsing, the parser consults the Trie to seamlessly resolve multi-word identifiers into standard `NODE_PHRASAL_CALL` function calls.

## 2. The Compiler Backend (Code Generation)
The backend walks the AST generated by the frontend and translates it into a flat stream of optimized bytecode instructions.

### 2.1 `compiler.c` / `compiler.h` (The Master Pipeline)
The entry point that bridges the frontend and backend.
* **The Compilation Flow**: Takes raw source code, passes it to `parseSource()`, intercepts the AST if `printAstFlag` is enabled, and feeds it into `generateCode()`.
* **Safety Net**: Returns `NULL` immediately if the parser flags any syntax errors, protecting the VM from executing malformed bytecode.

### 2.2 `codegen.c` / `codegen.h` (The Tree Walker)
"The Master Entry Point for the Back-End."
* **Recursive Walking**: Recursively traverses the AST, translating each `Node` into corresponding OpCodes.
* **The Loop State Tracker**: Maintains a `CompilerLoop` linked list to track `break` and `skip` (continue) statements, correctly patching jump offsets after a loop body has been fully traversed.

### 2.3 `emitter.c` / `emitter.h` (Compiler State & Scope)
Manages the low-level byte writing and lexical scope compilation.
* **Local Variables**: The `Compiler` struct maintains an array of `Local` structures to track variable scoping and stack slot indexing.
* **Bytecode Emission API**: Exposes helpers like `emitByte()`, `emitJump()`, and the recently optimized `emitGetLocal()`, taking care of `_LONG` instruction routing natively.
* **Scope Exiting**: Implements `endScope()`, smartly utilizing `OP_POP_N` to clean up multiple local variables simultaneously upon block exit.

### 2.4 `chunk.c` / `chunk.h` (The Instruction Set Architecture)
Defines the `Chunk` and the VM's OpCodes.
* **The ISA**: Contains the `OpCode` enum definitions (e.g., `OP_ADD`, `OP_GET_LOCAL_LONG`, `OP_PUSH_BYTE`).
* **The Bytecode Array**: The `Chunk` struct houses the raw `uint8_t` array of execution code, alongside a parallel `lines` array for stack traces.
* **The Constant Pool**: Handles deduplicated numbers and strings, unless bypassed by optimizations like `OP_PUSH_BYTE`.

## 3. The Virtual Machine (Runtime Execution)
The VM executes the compiled bytecode chunks using a high-performance interpreter loop.

### 3.1 `vm.c` / `vm.h` (The Core Engine)
The execution heart of the language.
* **The CallFrame Stack**: Maintains an array of `CallFrame` structs, each representing an actively executing function. Frames hold the Instruction Pointer (`ip`) and the base pointer for local variable slots.
* **The Computed Goto Dispatcher**: Uses a highly optimized GCC-specific computed goto table (`dispatchTable`) to jump directly from opcode to opcode, avoiding the overhead of massive `switch` statements.
* **The Value Stack**: Manages the flat array of `Value`s used for all mathematical and logical operations.

### 3.2 `value.c` / `value.h` (The Type System)
Defines the absolute primitive of the language: the `Value`.
* **NaN-Boxing**: Tricks the C Compiler utilizing Type Punning. All `Value`s are essentially IEEE-754 doubles.
* **Custom Type Tags**: Booleans, Nils, and Empty states hide inside the low bits of the NaN mantissa (`TAG_NIL`, `TAG_TRUE`), allowing primitives to exist without heap allocation.
* **Object Pointers**: Object pointers are encoded directly into the double's payload bits via bitwise masking.

### 3.3 `memory.c` / `memory.h` (The Garbage Collector)
The memory management oracle.
* **The Mark-and-Sweep GC**: Automatically triggers when `vm.bytesAllocated` exceeds `vm.nextGC`.
* **The Gray Stack**: Uses a worklist to traverse through objects, marking them black. It safely sweeps away any unreached white objects, freeing C memory.

### 3.4 `table.c` / `table.h` (The Hash Table Engine)
The core dictionary implementation.
* **Universal Keys**: The `Entry` struct accepts ANY `Value` as a key.
* **Linear Probing**: Implements an open-addressing hash table with linear probing and tombstone markers for seamless deletion.

### 3.5 `object.c` / `object.h` (Heap Allocations)
Defines complex reference types.
* **ObjKinds**: Strings, Lists, Dictionaries, Functions, and Native Functions.
* **String Interning**: Guarantees that every unique string exists exactly once in memory via FNV-1a hashing, allowing strings to be compared for equality by pointer reference alone (O(1) string equality).

## 4. Specialized Runtime Semantics
These modules offload complex operations from the main VM loop to keep `vm.c` clean.

* **`cast.c` / `cast.h`**: The universal router for type conversions. It safely attempts to cast any `Value` to an expected type (e.g., joining an integer into a string).
* **`subscript.c` / `subscript.h`**: Handles array/dictionary lookups (`list[1]`) and mutations (`dict["key"] = 5`), strictly enforcing Moon's 1-based indexing paradigm.
* **`error.c` / `error.h`**: THE SPELLCHECKING ORACLE. Uses Levenshtein distance to provide typo corrections when a reference error occurs. Also bridges errors into the LSP diagnostic array.

---

## 5. The Standard Library
Native C primitives injected directly into the VM environment at boot.

* **`lib_core.c`**: Exposes fundamental phrases like `show` and `ask`.
* **`lib_io.c`**: Empathetic OS interception for file reading/writing.
* **`lib_list.c`**: Core array manipulation (e.g., `reverse`).
* **`lib_math.c`**: Math wrappers over the C standard `<math.h>` (e.g., `sin`, `sqrt`).
* **`lib_string.c`**: String manipulations (`upper`, `lower`).

---

## 6. Developer Tooling

* **`lsp.c` / `lsp.h`**: The Language Server Protocol. Houses the infinite `listen/respond` loop to communicate with editors like NeoVim via JSON RPC. It utilizes its own "Semantic Symbol Table" to track variable types natively for autocomplete and hover data.
* **`cJSON.c` / `cJSON.h`**: The JSON parser utilized by the LSP.
* **`debug.c` / `debug.h`**: The Disassembler and Stack Visualizer used to verify opcode emission during development.
* **`main.c`**: Handles command-line arguments and implements the "Smart Prompt Calculator" for multi-line REPL block entries.
